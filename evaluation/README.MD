# Evaluation

This folder contains reusable scoring utilities for SLURP-TN ASR and SLU recipes.

The main script is:

```text
scoring.py
```

It computes corpus-level metrics from prediction files generated by the SpeechBrain recipes.

## Prediction file format

The expected input is a JSONL file where each line contains one prediction record:

```json
{"id": "slurptn_test_000000", "ref": "...", "hyp": "..."}
```

Required fields:

| Field | Description |
|---|---|
| `id` | Utterance identifier |
| `ref` | Reference text |
| `hyp` | Hypothesis text |

The structure must remain unchanged because the same files may be reused by other evaluation pipelines.

Example ASR record:

```json
{"id": "slurptn_test_000001", "ref": "بلاهي فيقني السبعة و النصف", "hyp": "بلاهي فيقني السبعة و النصف"}
```

Example SLU record:

```json
{"id": "slurptn_test_000002", "ref": "بلاهي فيقني <time> السبعة و النصف متاع الصباح >", "hyp": "بلاهي فيقني <time> السبعة و النصف متاع الصباح >"}
```

## Metric backend

The scoring script uses [JiWER](https://github.com/jitsi/jiwer) for WER and CER computation.

Install it with:

```bash
pip install jiwer
```

or through the repository requirements:

```bash
pip install -r requirements.txt
```

## Corpus-level scoring

All metrics are computed at the dataset level.

This means the script does **not** compute scores sentence by sentence and then average them.

Instead, JiWER receives the complete list of references and hypotheses:

```python
jiwer.wer(refs, hyps)
jiwer.cer(refs, hyps)
```

where:

```python
refs = [ref_1, ref_2, ref_3, ...]
hyps = [hyp_1, hyp_2, hyp_3, ...]
```

This gives corpus-level WER and CER.

## ASR metrics

For ASR, references and hypotheses are plain transcriptions.

The ASR task computes:

- WER
- CER

Example:

```text
REF: بلاهي فيقني السبعة و النصف
HYP: بلاهي فيقني السبعة و النصف
```

The scorer computes:

```text
WER
CER
```

directly on the reference and hypothesis text.

## SLU annotation format

The SLU recipes use inline slot annotations.

Example:

```text
بلاهي فيقني <time> السبعة و النصف متاع الصباح >
```

In this format:

```text
<time> السبعة و النصف متاع الصباح >
```

is one annotated span.

The concept label is:

```text
time
```

The concept value is:

```text
السبعة و النصف متاع الصباح
```

## SLU metrics

For SLU, the scorer computes:

- WER
- CER
- CoER
- CVER

### WER and CER for SLU

For SLU, WER and CER are computed after removing the slot tags while keeping the spoken words.

Example:

```text
بلاهي فيقني <time> السبعة و النصف متاع الصباح >
```

becomes:

```text
بلاهي فيقني السبعة و النصف متاع الصباح
```

Then JiWER computes corpus-level WER and CER.

### CoER

CoER means **Concept Error Rate**.

It is computed as WER over the sequence of concept labels.

Example:

```text
REF: بلاهي فيقني <time> السبعة و النصف متاع الصباح >
HYP: بلاهي فيقني <date> السبعة و النصف متاع الصباح >
```

Concept sequences:

```text
REF: time
HYP: date
```

This is an error for CoER.

If the concept label is correct but the value is wrong:

```text
REF: <time> السبعة و النصف متاع الصباح >
HYP: <time> الثمانية متاع الصباح >
```

then CoER does not count the value mismatch, because both concept labels are:

```text
time
```

### CVER

CVER means **Concept-Value Error Rate**.

It is computed as WER over concept-value units.

Each annotated span is treated as one unit.

Example:

```text
<time> السبعة و النصف متاع الصباح >
```

becomes one concept-value token:

```text
time=السبعة_و_النصف_متاع_الصباح
```

The underscores are used internally so that the full concept-value pair is treated as one token by JiWER.

This means:

```text
REF: <time> السبعة و النصف متاع الصباح >
HYP: <time> السبعة و النصف متاع الصباح >
```

is correct for CVER.

But:

```text
REF: <time> السبعة و النصف متاع الصباح >
HYP: <date> السبعة و النصف متاع الصباح >
```

is wrong because the concept label changed.

And:

```text
REF: <time> السبعة و النصف متاع الصباح >
HYP: <time> الثمانية متاع الصباح >
```

is also wrong because the concept value changed.

## Whisper special tokens

The scorer removes Whisper-style special tokens such as:

```text
<|ar|>
<|transcribe|>
<|notimestamps|>
```

These are removed before metric computation.

SLU tags such as:

```text
<time>
<date>
>
```

are preserved for SLU parsing.

## Using the scorer from Python

Example for ASR:

```python
from Evaluation.scoring import compute_metrics_from_jsonl

metrics = compute_metrics_from_jsonl(
    predictions_file="results/whisper_base_asr/test_scores/predictions.jsonl",
    task="asr",
)

print(metrics)
```

Output:

```python
{"WER": 12.34, "CER": 4.56}
```

Example for SLU:

```python
from Evaluation.scoring import compute_metrics_from_jsonl

metrics = compute_metrics_from_jsonl(
    predictions_file="results/whisper_base_slu/test_scores/predictions.jsonl",
    task="slu",
)

print(metrics)
```

Output:

```python
{"WER": 12.34, "CER": 4.56, "CoER": 8.90, "CVER": 10.11}
```

## Output metrics file

The recipes write a `metrics.txt` file next to the prediction file.

Example:

```text
WER: 12.34
CER: 4.56
CoER: 8.90
CVER: 10.11
```

For ASR, only WER and CER are written.

For SLU, WER, CER, CoER, and CVER are written.

## Compatibility with external evaluation

The JSONL prediction files are kept simple on purpose.

They can be converted later to formats required by other tools such as SCTK/SCLITE.

The scoring helper does not modify the JSONL structure.
